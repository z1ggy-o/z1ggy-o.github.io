Python Multithreading 退出的小问题

z1ggy's blog
2019-05-03 (Updated: 2019-05-03)
Python
Python

最近在做 socket 编程作业的时候遇到了一个小问题。作业要求在 console 输入 Ctrl-C 之后程序能够正常处理好 socket 的关闭，最后再正常退出。思路上就是常见的，先 catch signal 之后再用相应的 handler 部分来做资源关闭。但尝试了几个方法之后，发现在给入 signal 之后，要么程序停住，等待 client 侧断开；要么 main thread 没有退出。

这个问题的重点不在于 socket，而是在 main thread 和 child thread 都在无法自我结束的情况下，如何接受信号，退出程序。下面先讲讲我的 「错误」 思路。

我的 「错误」 思路

首先，server 侧的程序逻辑是，main tread 开启 server socket 等待连接，在接到连接之后，将 connection socket 交给 child thread 去进行处理。Pseudocode 如下：

Python 已经有一个比较高级的 module 叫做 threading。它是对大家已知 thread 的一个高级扩展，提供了一些方便的 method：例如返回所有 thread，返回 main thread 等等。在看了 threading 的文档之后，构想了三个方案：

  1. 像其他 multithread 编程一样，接到信号后让各个 thread 自己处理完自己的事务，利用 join() 去等待 child thread 全部完成自己的任务后，退出程序。
  2. 利用一个 global variable，将其作为 flag 放到 run() 中的 while 去。当我们改变这个 variable 的时候，child thread 就能从 while loop 中脱出，从而结束。
  3. 将 child thread 设定为 daemon thread，这样在 main thread 这个 non daemon thread 结束之后，daemon thread 也被结束，从而退出整个程序。

问题点

join() 带来的问题

第一个方案是绝对行不通的，因为 child thread 因为 while loop 是处于 busy 状态的，在 join 之后 main thread 被 block，child thread 又不能退出，成了一个死局。

第二个方案是基于第一个方案的漏洞做的改进。在用一个 global variable 作为控制 while loop 的 condition 之后，我们就可以等待 child thread 正常退出了。如果你的 child thread 内部没有其他阻碍退出的逻辑，那么这个方式是很好的。因为我们可以按照自己想要的方式去处理好打开的资源。

但是在我自己的这个情况下，这个方法也是不行的。因为 socket.recv() 会将 thread block，等待 client 侧的 message。我这里还没有设置 timeout，所以就算 while loop 结束了，child thread 还是被 block。这也是为什么，等到 client 再次发送信息之后，server 侧才能退出的原因。

还有一个方法就是利用 os._exit() 而非 sys.exit() 退出。os._exit() 会直接将程序关闭，不做任何其他的处理。一般用于 child thread 关闭程序。

setDaemon()

那么第二个方案问题在哪里呢？第二个方案其实没有问题，我在退出 main thread 之前利用 threading 提供的机能处理好了各个 child thread 的 client sockets，最后退出程序。

那么哪里出了问题呢？语法。
在 Python 的文档中提到，setDaemon() 是一个已经过时的 method，让大家把它当做一个 property 使用就好了。所以一开始我使用了 setDaemon = True 来进行设定，没有报错，但是实际上没有被设为 daemon。还是应该使用 setDaemon(True) 这个语法。也可能是有其他的 property 名，一下子没找到，就懒得查了。

使用 daemon thread 的时候得想好如何处理打开的资源。因为 non daemon thread 一旦全部退出，daemon thread 也就随之关闭了。
